「你改的是 Harness，不是模型，但出问题的地方往往在这里。」这句话出现在 2026 年春天的某场技术评审里，说完之后全场安静了两秒。

评审者的意思很清楚：模型的权重没动，LLM 没有任何变化，但排名从 30 名开外跃升到第 5 名的 TerminalBench 2.0 成绩，全靠 Harness 的改动。

![](https://iili.io/Bnnm9UJ.png)
> 这一改，边界就开始漂了

换句话说，真正值得测的不是模型本身，而是让模型真正有用的那层「机械装置」。

这个事实构成了整个专题的起点。如果你准备的是 AI Agent 相关的实习或校招面试，你会发现几乎每轮技术面都会绕到同一个问题：「你测的是什么？

」或者更具体一点：「你怎么确保你的改动没有破坏 Agent 的核心行为？」能答清楚的人不多。

不是因为这个问题有多深，而是因为大多数教材把「测试 LLM 输出」和「测试 Harness 逻辑」混在一起讲，让候选人分不清边界，一被追问就露馅。

还有一个数字值得记住：10 步流程，每一步的成功率是 99%，端到端的成功率只剩约 90.4%。这是复利式的失败复合，每个 Agent 工程师都逃不掉这个算术题。

所以测 Harness 不仅是「写几个单元测试」这么简单，它本质上是在问：你有没有能力在错误复合之前把它截住。

这篇专题从四个维度把这个问题的答案拆开：先说清概念边界，再列出高风险改动路径，然后给出生产级验证循环的设计方法，最后回到回归测试体系的完整构建。

每一节都对着「面试能说什么、项目能怎么写、追问能怎么接」来写。

## 一、先把概念说清楚：什么是 Agent loop 核心逻辑，什么是 Harness

### 1.1 Agent 与 Harness 的边界

在真正开始讲测试策略之前，必须把 Agent 和 Harness 的边界说死。面试里有一道经典追问：「你做的是一个 Agent，还是一个让模型变成 Agent 的东西？

」能接住这个问题的人，通常在概念层面就没有混淆这两个东西。

Beren Millidge 在 2023 年的《Scaffolded LLMs as Natural Language Computers》里给了一个精准的类比：原始 LLM 就像一个没有内存、没有硬盘、没有 I/O 的 CPU。

上下文窗口充当 RAM（快但有限），外部数据库充当硬盘（大但慢），工具集成相当于设备驱动，而 Harness 就是操作系统。

[millidge_scaffold] 这个类比的好处是，它把一个抽象概念落实成了每个人在操作系统课上都见过的图景：CPU 本身不生产智能，智能靠操作系统调度资源、读写内存、执行指令来呈现。

用更工程化的语言来说：Agent 是涌现出来的行为——那个有目标、会用工具、能自我纠错的实体，用户看到的就是它。Harness 是产生这种行为的「机械装置」。

[agentharness_blog] 这两个东西不是同一个层次。当有人说「我做了一个 Agent」，实际意思是他做了一个 Harness，然后把它指向了一个模型。

你测的到底是模型的能力，还是 Harness 的可靠性，决定了你对问题的理解深度。

Anthropic 的 Claude Code 文档里直接把这个边界说得很清楚：SDK 就是「驱动 Claude Code 的 Agent harness」。

[agentharness_blog] OpenAI 的 Codex 团队也用了同样的表述，明确把「Agent」和「Harness」等同起来，指代让 LLM 真正有用的非模型基础设施。

换句话说，在头部公司的官方文档里，这个概念已经被明确命名了，不是社区的发明创造，是工程实践的共识。

所以面试里被问到「什么是 Agent Harness」，你完全有底气直接引用这个表述，不需要绕弯子。

![](https://iili.io/B4bJYB4.png)
> 官方文档直接命名，这题答案不用自己编

### 1.2 Harness 的 12 个组件速览

一个生产级的 Agent Harness 并不是一个单一模块，它是一组协同工作的组件。

根据 Anthropic、OpenAI、LangChain 和更广泛的实践者社区的经验，生产级 Harness 有 12 个独立组件：[agentharness_blog]

**编排循环（Orchestration Loop）** 是心脏，实现思考-行动-观察（TAO）循环，也叫 ReAct 循环。

循环运行机制是：组装提示词、调用 LLM、解析输出、执行工具调用、反馈结果、重复直到完成。机械上看通常就是一个 while 循环，复杂性来自循环管理的所有东西，而不是循环本身。

Anthropic 把他们的运行时描述为「哑循环」——所有智能都在模型里，Harness 只管理回合，这个表述对测试设计有根本性的影响，后面会展开。

**工具（Tools）** 是 Agent 的手。工具被定义为模式（名称、描述、参数类型），注入到 LLM 的上下文中，让模型知道有什么可用。

工具层处理注册、模式验证、参数提取、沙盒执行、结果捕获，以及把结果格式化回 LLM 可读的观察。

Claude Code 提供六类工具：文件操作、搜索、执行、网络访问、代码智能和子智能体生成。

OpenAI 的 Agents SDK 支持函数工具（通过 @function_tool）、托管工具（WebSearch、CodeInterpreter、FileSearch）和 MCP 服务器工具。

**记忆（Memory）** 分短期和长期。短期记忆是单个会话内的对话历史，长期记忆跨会话持久化。

Claude Code 实现了三层层次结构：轻量级索引（每条约 150 字符，始终加载）、按需拉取的详细文件，以及仅通过搜索访问的原始记录。

Anthropic 使用 CLAUDE.md 项目文件和自动生成的 MEMORY.md 文件，LangGraph 使用命名空间组织的 JSON 存储，OpenAI 支持由 SQLite 或 Redis 支持的 Sessions。

**上下文管理（Context Management）** 是许多 Agent 失败的地方。

Chroma 的研究发现，当关键内容落在上下文窗口中间位置时，模型性能下降 30% 以上，这就是「Lost in the Middle」效应。斯坦福的同名研究也证实了这一点。

即使是百万 token 窗口，随着上下文增长，指令遵循能力也会退化。

生产级处理策略包括压缩（在接近限制时总结对话历史）、观察遮蔽（隐藏旧的工具输出但保持工具调用可见）、即时检索（维护轻量级标识符，动态加载数据）和子智能体委托（每个子智能体广泛探索，但只返回压缩摘要）。

[anthropic_three_agent]

剩余组件包括：**提示词构建**（分层组装系统提示词、工具定义、记忆文件、对话历史和当前用户消息）、**输出解析**（现代 Harness 依赖原生工具调用，模型返回结构化的 tool_calls 对象）、**状态管理**（LangGraph 将状态建模为流经图节点的类型化字典，用 reducer 合并更新）、**错误处理**（LangGraph 区分四种错误类型：瞬态、LLM 可恢复、用户可修复和意外）、**防护栏和安全**（OpenAI 实现三个级别：输入、输出和工具防护栏）、**验证循环**（Anthropic 推荐三种方法：基于规则的反馈、视觉反馈和 LLM 评判者）和**子智能体编排**（Claude Code 支持 Fork、Teammate、Worktree 三种执行模型）。

这 12 个组件里，哪些改动最可能破坏 loop 核心逻辑？答案是前三项——编排循环、状态管理和上下文管理，以及验证循环本身。

它们要么直接影响控制流走向，要么决定信息是否正确传递到下一个节点，要么决定错误是否被及时截住。后续章节会逐个拆解这四个高风险路径。

### 1.3 「哑循环」的正确理解

Anthropic 把他们的运行时描述为「哑循环」，所有智能都在模型里，Harness 只管理回合。这个表述为什么重要？因为它直接影响测试策略的设计方向。

如果 Harness 是「哑」的，意味着测试不应该围绕「模型是否做出了正确的推理」来设计，而应该围绕「Harness 是否正确传递了输入、正确执行了工具调用、正确捕获了输出、正确更新了状态」来设计。

换句话说，测试的对象是 Harness 的确定性逻辑，而不是模型的不确定性输出。

这个区别在面试里经常被考到。面试官会问：「你怎么决定一个测试应该通过还是失败？」如果你回答「看模型的输出是否正确」，就暴露了你对 Harness 和 Agent 边界的混淆。

正确答案应该是：Harness 的测试是确定性的，它验证的是编排逻辑、状态传递、错误处理是否按预期运行；

模型的不确定性应该在 Harness 的验证循环里去处理（比如用 LLM 评判者），而不是成为单元测试的一部分。

这个思路在实际工程里也有对应。Claude Code 的创建者 Boris Cherny 指出，给模型一种验证其工作的方法，质量能提高 2 到 3 倍。

[agentharness_blog] 这句话的潜台词是：验证步骤让模型学会自我纠错，而 Harness 的任务是提供这种验证手段，而不是替模型做判断。

![](https://iili.io/BiNA3aR.png)
> 模型跑偏了，到底是 Harness 没拦住还是模型本身的问题

## 二、Harness 改动的四个高风险路径

知道了 Harness 是什么，接下来要看改哪里最容易把 loop 改死。这四个高风险路径不是教科书式的分类，而是从真实翻车案例里提炼出来的规律。

### 2.1 编排循环层：条件边判断逻辑被改写

编排循环是 Agent 的心脏，每一次「下一步该做什么」都经过它。如果条件边的判断函数出了 bug，Agent 的行为会立刻偏离预期。

最典型的翻车场景是这样的：你维护一个基于 LangGraph 的客服 Agent，正常流程是：用户提问 → Agent 调用知识库搜索工具 → 根据搜索结果决定是否继续追问还是直接回复。

判断函数 `should_continue` 控制着这条分支：

```python
def should_continue(state: MessagesState) -> Literal["tools", END]:
    messages = state['messages']
    last_message = messages[-1]
    # 如果 LLM 进行工具调用，路由到 tools 节点
    if last_message.tool_calls:
        return "tools"
    # 否则停止（回复用户）
    return END
```

某天你对这个函数做了一次「优化」——因为发现某些情况下 LLM 返回的不是 tool_calls，而是 tool_call_id 字段，你加了一个 or 条件。

然后线上开始出现「用户问问题，Agent 直接回复了错误答案，根本没调搜索工具」的投诉。原因是新加的 or 条件优先级写错了，关键路径被错误地短路了。

这个问题在 LangGraph 的 conditional_edges 机制里特别容易出现。

LangGraph 官方的 StateGraph 使用条件边来决定节点间的路由，路由矩阵覆盖所有状态组合是确保改动不破坏逻辑的核心手段。

[langgraph_docs] 测试策略应该是：构建「所有状态 → 目标边」的路由矩阵，然后逐个验证每个状态组合是否映射到正确的目标边。

具体来说，你需要列出所有可能的 `state['messages']` 状态：

- 空消息列表

- 只有 HumanMessage

- 有 HumanMessage + AssistantMessage（无 tool_calls）

- 有 HumanMessage + AssistantMessage（有 tool_calls）

- 有 HumanMessage + 多个 AssistantMessage

- 有 ToolMessage（工具调用失败返回的错误消息）

然后验证 `should_continue` 对每种状态都返回正确的边。

LangGraph 官方在 [langgraph_docs] 里明确建议用 `add_conditional_edges` 时附带路由函数的完整覆盖测试，因为条件边是控制流的第一个守门人。

![](https://iili.io/B6pAHPt.png)
> 改了个 if else 优先级，线上就开始批量返回空结果

### 2.2 状态管理层：Reducer 合并规则被破坏

LangGraph 的 State 更新机制是另一个高频翻车点。每次节点执行后，返回值通过 reducer 函数合并到主状态。

如果 reducer 合并规则被破坏，关键字段在跨节点传递时会悄悄消失或被覆盖，而且不会抛异常，只会让 Agent 表现出「行为越来越奇怪但日志看起来正常」。

LangGraph 将状态建模为流经图节点的类型化字典，用 reducer 合并更新。检查点发生在超级步骤边界，支持中断后恢复和时间旅行调试。

[agentharness_blog] 这个机制的好处是状态传递透明，坏处是如果你在某个节点里不小心把整个 state 覆盖了而不是合并，之前的上下文就会静默丢失。

举一个具体场景：你的 Agent 有两个节点——`agent` 节点负责调用 LLM，`tools` 节点负责执行工具调用。

`agent` 节点返回 `{"messages": [response]}`，正常情况下这个更新会把新的 assistant 消息追加到 messages 列表末尾。

如果你的 reducer 配置错误，用的是「覆盖」而不是「追加」，messages 列表就会被替换成只有一条消息，对话历史全部丢失。

用户看起来是「Agent 每次只记得最近一次对话」，但这不是模型的 bug，是 Harness 的状态管理问题。

测试策略必须覆盖两个层面：每个节点的输入/输出 schema 测试，以及 reducer 合并规则测试。具体做法是：

对于 schema 测试，验证每个节点在各种输入状态下是否返回了预期的字段类型和结构。

比如 `agent` 节点必须返回包含 `messages` 字段的字典，`tools` 节点在工具调用成功时返回什么、失败时返回什么，都要有明确的 schema 断言。

对于 reducer 合并规则测试，要验证跨节点传递时关键字段是否保持一致。

比如 messages 列表在经过 agent → tools → agent 三个节点后，内容是否正确累加，是否有字段被意外覆盖或清空。

LangGraph 支持自定义 reducer，如果你修改了默认的列表 append reducer 为自定义的「去重 reducer」，就必须重新跑覆盖所有状态组合的测试。

Anthropic 的三 Agent 架构设计里也特别提到了状态管理的重要性。

他们的实现中，context resets 配合结构化 handoff artifacts，让下一个 Agent 在工作流中能从定义好的状态继续。

这种设计思路对测试的启发是：状态传递必须有显式的 schema 契约，而不是靠「约定」来维持一致。[anthropic_three_agent]

### 2.3 上下文管理层：压缩策略误删推理轨迹

「Lost in the Middle」效应是 Agent 工程里最容易被忽视但影响最大的问题之一。当关键内容落在上下文窗口中间位置时，模型性能下降 30% 以上。

[agentharness_blog] 这不是 LLM 的设计缺陷，而是人类注意力机制在上下文窗口上的投射——无论是人类读者还是 LLM，对「开头和结尾」的记忆都优于「中间」。

上下文压缩策略是解决这个问题的常见手段，但也是高风险改动点。

最典型的翻车案例：你的 Agent 在处理长文档时，使用一个简单的压缩函数——当 messages 列表超过 20 条时，删除最早的三分之一。逻辑看起来合理：腾出空间给新消息。

但这个策略的问题在于，推理链中的关键中间结果往往不在最早的三分之一，也不在最后的三分之一，而在中间。

压缩函数把推理链「为什么 A 导致 B」的关键步骤删掉了，Agent 在下一轮推理时发现自己「没有理由得出这个结论」，行为开始漂移。

Claude Code 的压缩策略提供了一个正面参考：它保留架构决策和未解决的 bug，丢弃冗余的工具输出（即那些 LLM 已经处理过的工具返回值的原始文本）。

[agentharness_blog] 这个取舍的核心逻辑是：压缩的目标不是最大化 token 减少，而是最大化「保留对下一轮推理有价值的信号」。

测试压缩策略的正确方式，是验证**压缩前后的任务完成率**，而不是压缩率数字本身。

如果你的压缩策略把上下文从 8000 token 压缩到 2000 token，看起来压缩率 75% 很漂亮，但任务完成率从 95% 掉到了 60%，这个压缩策略就是失败的。

测试用例应该覆盖：

1. **压缩前的任务通过基准**：用未压缩的完整上下文跑同样的任务集，记录通过率作为基准。

2. **压缩后的任务通过率**：用压缩后的上下文跑同样的任务集，对比基准是否有显著下降。

3. **边界条件**：验证「恰好在上下文窗口边界」的任务是否正常处理，压缩触发时机是否稳定。

Anthropic 的上下文工程指南说明了目标：找到最小的高信号 token 集合，最大化期望结果的可能性。

[agentharness_blog] 这个目标不能用简单的「压缩率」来衡量，必须用「任务完成率」来衡量——这是一个面试里容易被问到但很多人答不清楚的点。

ACON 研究显示，通过优先考虑推理轨迹而不是原始工具输出，token 减少 26% 到 54%，同时保持 95% 以上的准确性。

[agentharness_blog] 这组数据值得记住，因为它给出了压缩策略有效性的量化参考线：token 减少 26% 到 54% 且保持 95% 以上准确率，是当前工程里可接受的范围。

![](https://iili.io/BiiNQkB.png)
> 上下文窗口剩 10%，Agent 开始随机回复了

### 2.4 验证循环层：纠错阈值被放松

验证循环是区分「玩具演示」和「生产级 Agent」的关键。

Anthropic 推荐三种验证方法：基于规则的反馈（测试、linter、类型检查器）、视觉反馈（通过 Playwright 截图用于 UI 任务）和 LLM 评判者（单独的子 Agent 评估输出）。

[agentharness_blog] 但验证循环本身也是一个可以被改坏的地方。

最常见的翻车场景是阈值被放宽。假设你的验证循环里有一个关键操作检查：验证 Agent 是否正确调用了数据库写入工具。初始阈值是 0.01%——几乎不允许任何漏检。

但在某次「优化」里，因为发现误报率有点高，阈值被放宽到 2%。

然后线上开始出现「Agent 执行了删除操作但验证通过了」的投诉，因为 2% 的漏检率在高频调用场景里意味着每小时都有几次漏检。

Stripe 的生产实践是把重试上限设为两次，这是为数不多的可以直接引用的生产级数据锚点。

[agentharness_blog] 两次重试加阈值 0.01% 的组合，构成了一个具体的、可对比的参考。

但更重要的是，Stripe 的实践说明了「可接受漏检率」是一个场景依赖的决策，而不是一个固定数字。关键操作（删除、发送、支付）的阈值趋近 0%，信息查询可以放宽到 1%-2%。

测试策略应该区分两类操作：对关键操作（高风险操作）使用严格的阈值，对信息查询操作使用宽松的阈值。具体来说：

**关键操作验证**：每次工具调用都检查返回值状态。

如果工具调用的语义是「修改外部状态」——写文件、删数据库、发送消息、调用支付接口——验证循环必须以接近 100% 的精度截住错误。

任何大于 0.1% 的漏检率都应该触发告警和人工 review。

**信息查询验证**：搜索、读取、分析类操作的验证可以适度放宽，因为错误的影响范围有限，可以通过重试自然修复。验证循环的任务是确保「至少拿到了一次有效结果」，而不是「每次都拿到最优结果」。

这个区分在面试里是高频追问点：「2% 的漏检率能接受吗？」如果你能给出「取决于使用场景，关键操作趋近 0%，信息查询可放宽」这个判断框架，就已经超过了大多数候选人的回答水平。

## 三、生产级验证循环设计：不让改动偷偷穿透防线

### 3.1 三种验证范式的对比与取舍

上一节提到了三种验证方法：规则测试、视觉反馈和 LLM 评判者。但面试官真正想追问的不是「有哪些验证方法」，而是「你如何选择，不同选择的取舍是什么」。

| 验证方式 | 确定性 | 延迟 | 适用场景 |
|---|---|---|---|
| 规则测试（测试/linter） | ✅ 高 | 低 | 工具调用格式、字段 schema、退出条件 |
| 视觉反馈（Playwright 截图） | 中 | 中 | UI 任务输出正确性、渲染结果验证 |
| LLM 评判者 | 低 | 高 | 语义正确性、流畅性、生成质量 |

规则测试的确定性最高，延迟最低。

工具调用的格式是否正确、返回的 schema 是否匹配、是否满足退出条件（`should_continue` 返回 END），这些都可以用确定性规则来验证。

比如用 JSON schema 验证工具调用的参数结构，用正则表达式验证输出格式，用布尔断言验证字段值范围。

这类测试可以集成到 CI 里，每次 PR 都跑，没有 flaky 的问题。

视觉反馈适合需要验证「渲染结果是否正确」的场景，比如 Agent 生成了一段代码，你要验证这段代码在目标环境下是否能正常运行。

Anthropic 的三 Agent 框架里，evaluator agent 通过 Playwright MCP 导航到实时页面，交互界面并提供详细评估，这是一种结构化的视觉反馈实现。

[anthropic_three_agent] 在面试里提到这个案例，说明你对验证循环有工程级别的理解，而不只是知道「可以截图对比」。

LLM 评判者是不确定性最高的验证方法，但也是处理「没有确定性 ground truth」场景的唯一手段。

生成一段回复的流畅性、一段解释是否有幻觉、一次对话是否自然——这些问题没有规则可以穷举，必须靠另一个 LLM 来评判。

但 LLM 评判者有两个问题：一是延迟高，每次验证都要多跑一次 LLM 调用；二是评价不稳定，同样的输出可能被不同的模型版本给出不同的分数。

Artem Bredikhin 在 LinkedIn 上写过一句值得记住的话：长期运行的 AI Agent 失败的原因很简单——每个新的上下文窗口都是一次失忆。

突破在于结构化：JSON feature specs、强制测试、按提交逐步推进，以及确保每次会话都从可工作的应用开始的初始化脚本。

[anthropic_three_agent] 这句话指向了一个关键洞察：验证循环的设计不是「堆更多检查点」，而是「让每次检查点都能恢复状态」。

### 3.2 验证循环与循环本身的协同设计

Martin Fowler 的 Thoughtworks 团队把验证方法框架化为两类：前馈（feedforward，行动前引导）和传感器（sensor，行动后观察）。

[agentharness_blog] 这个区分对 Harness 设计很重要，因为它决定了验证发生在循环的哪个位置。

**前馈验证**发生在行动执行之前。比如在调用工具之前，检查参数是否符合 schema。在 Agent 做出决策之前，检查当前状态是否满足某个前置条件。

这相当于在代码里加断言——提前截住明显错误，不让它们进入执行阶段。

**后馈验证**发生在行动执行之后。比如工具调用返回后，验证返回结果是否有效。Agent 给出了最终回复后，用 LLM 评判者检查回复质量。

这相当于测试驱动开发里的「先跑测试再看结果」——先执行，再判断是否符合预期。

Claude Code 的 collect → act → verify 循环是这个协同设计的正面案例。

收集上下文（搜索文件、读代码）对应前馈验证，行动（编辑文件、运行命令）对应执行，验证结果（运行测试、检查输出）对应后馈验证。

关键是这个循环不是单向的——验证失败会直接反馈到下一个 collect 阶段，让 Agent 在下一轮推理里调整策略。

[agentharness_blog] 换句话说，验证步骤让模型学会自我纠错，而不是让模型犯完错就停在那里。

这个模式在面试里可以直接落到项目说法：「我在项目里实现了 collect → act → verify 三阶段循环，验证失败时会自动重试，最大重试次数是两次，超过后触发告警并保留现场日志供人工排查。

」这段话说清楚了三个要素：循环结构、失败策略和可观测性。

Claude Code 的创建者 Boris Cherny 指出，给模型一种验证其工作的方法，质量提高 2 到 3 倍。

[agentharness_blog] 这组数据的面试价值在于，它给「要不要加验证循环」这个决策提供了一个量化依据：加验证循环的 ROI 大约是 2-3 倍质量提升，值得投入。

### 3.3 仿真环境的构造方法

验证循环设计好了，但这些验证必须在真实或接近真实的环境里跑才能发挥作用。仿真环境的构造是 Harness 测试里最考验工程能力的一环。

三类必须覆盖的边界 case：

**工具调用失败**。工具会失败，但 Agent 不应该因为工具失败就停下来。仿真环境需要能够模拟工具返回各种错误：超时、网络不可达、权限拒绝、参数校验失败、无返回结果。

在 LangGraph 里，工具调用失败会作为错误结果返回，模型需要拿到这个错误信息并决定下一步行动——是重试、降级还是放弃。

[langgraph_docs] 测试用例应该覆盖：单次失败（应重试）、连续失败（应降级或告警）、部分失败（部分工具成功，部分失败，Agent 是否正确处理）。

**上下文窗口逼近**。在接近上下文窗口限制时，Agent 的行为是否正常？压缩策略是否生效？

是否有信息被误删？这需要在仿真环境里构造「恰好在上下文窗口边界」的任务。

比如构造一个 15 步的对话历史，前 10 步是普通信息查询，后 5 步包含关键决策信息，总 token 数接近窗口限制的 90%。

然后验证 Agent 的最终决策是否正确，以及压缩日志里是否保留了后 5 步的关键信息。

**防护栏绊线触发**。

OpenAI 的 SDK 实现三个级别的防护栏：输入防护栏（在第一个 Agent 上运行）、输出防护栏（在最终输出上运行）和工具防护栏（在每次工具调用时运行）。

[agentharness_blog] 仿真环境需要能触发这些防护栏的边界条件，比如输入包含敏感词、输出包含潜在有害内容、工具调用试图访问受限资源。

验证循环应该确保防护栏绊线触发时，Agent 能安全停止而不是绕过限制继续执行。

Anthropic 在架构上将权限执行与模型推理分离：模型决定尝试什么，工具系统决定允许什么。

这是防护栏设计的一个重要原则——不要依赖模型来遵守安全约束，而要在 Harness 层强制执行。

Claude Code 独立管理约 40 个离散工具能力，分三个阶段管理权限：项目加载时建立信任、每次工具调用前权限检查、高风险操作的明确用户确认。

## 四、回归测试体系：从单步校验到端到端覆盖

### 4.1 单元测试层

单元测试是回归测试体系的第一道守卫。它的目标是验证每个 Harness 组件在隔离状态下是否按预期运行。

**每个 Node 的输入/输出 schema 测试**。在 LangGraph 里，每个节点是一个 Python 函数，接收 state 字典，返回更新后的 state 字典。

单元测试应该验证：给定一个特定的输入 state，节点是否返回了预期结构和值的输出 state。

比如 `agent` 节点的测试：输入是 `{"messages": [HumanMessage(content="今天天气如何")]}`, 输出应该是 `{"messages": [..., AssistantMessage(content=..., tool_calls=[...])}

### 5.1 课程 / 实验室场景的降级叙事

课程项目最大的问题不是项目太小，而是表达方式太像「学生作品展」。

比如你说「我做了一个多轮对话机器人」——这句话在面试官耳朵里听到的是「我调用了一个 API，加了历史记录，结束」。不是这个项目没有技术含量，而是你没有把它翻译成工程语言。

降级叙事的核心是把「我做了一个聊天机器人」翻译成「我实现了一个多轮对话状态机，用 LangGraph StateGraph 建模，每个节点负责特定意图识别和响应生成，状态通过 reducer 函数在节点间合并传递」。

这样表达的效果是完全不同的。前者是产品描述，后者是工程实现。

具体到测试策略，课程项目的降级叙事可以这样组织：

「在课程项目里，我为多轮对话状态机实现了状态更新逻辑，测试中遇到的一个关键问题是：当用户连续快速输入多条消息时，中间状态在节点间传递时会被部分覆盖。

具体来说，`should_continue` 判断函数在某些边缘条件下返回了错误的路由目标，导致对话在应该调用工具时直接跳到了 END。

我通过两种方式处理：一是为 `should_continue` 函数补充了 8 个边界 case 的单元测试，覆盖空消息、工具调用失败、超长输入等场景；

二是增加了重试机制，单次工具调用失败时最多重试两次，两次都失败时将错误状态写入日志并降级响应。测试覆盖率从原来的 45% 提升到了 78%。」

这段话交代了四个关键信息：遇到了什么问题、为什么这是个问题、具体怎么修的、结果怎么样。有问题、有分析、有行动、有量化，这就是工程级别的表达。

### 5.2 开源参与的精准说法

如果你参与过 LangChain、CrewAI 或者其他 AI 框架的贡献，不管是修 bug、加测试还是改进文档，在面试里都是加分项。

但「参与开源」这四个字本身没有信息量。面试官想听到的是：你发现了什么问题、为什么这是个问题、你设计了什么方案、后续如何避免同类问题。

精准说法的标准句式：「我在 LangChain 项目的测试模块里，为工具调用失败路径补充了 15 个边界 case，覆盖了超时、网络不可达、权限拒绝、工具返回空结果四种典型失败模式。

每个 case 验证了当工具返回特定错误时，Harness 是否正确进入重试、降级或告警状态，同时 reducer 函数处理错误状态时不会污染主状态。

这个测试套件后来被合并到项目的 CI 流程里，每次 PR 都会跑这组回归测试。」

如果你没有具体的开源贡献经历，但读过某个框架的源码并做过分析，同样可以讲。

重点是说清楚：「我读过 LangGraph 的 State 更新机制源码，理解了 reducer 函数如何在不同节点间合并状态，具体实践中发现了一个边界 case——当两个节点同时更新同一个字段时，后执行的节点会完全覆盖前者的更新，而不是合并。

这个问题在单 Agent 场景下不常出现，但在多 Agent 协作时会触发。我为这个场景设计了一个测试用例并在社区里提交了 issue。」

这里的核心是展示你不是在「用框架」，而是在「理解框架」。

### 5.3 实习局部贡献的放大叙事

有实习经历但没做过完整 Agent 项目的候选人，最常见的错误是只说「我加了测试」而不说「测了什么、结果怎么样」。

放大叙事的关键是量化。但量化不是凭空编数字，而是把已有的工作用正确的维度表达出来。

「在实习期间，我为组里的 Agent 项目补充了约 40 个回归测试 case，覆盖工具调用失败、状态管理和上下文管理三个高风险模块。

其中工具调用失败的测试 case 包括：单次超时、连续三次超时、权限拒绝、无返回结果四种场景，验证了重试策略是否按预期触发、降级机制是否正确执行、告警是否在超限后及时发出。

单元测试覆盖率从 45% 提升到 78%，在集成测试阶段发现了 3 个之前漏测的边界 case，其中一个已经在预发环境触发过线上告警。」

这段话的逻辑结构是：工作范围 → 具体 case → 覆盖场景 → 结果量化。信息密度足够，面试官可以随时打断追问具体细节。

如果你的实习确实没有机会接触 Agent 测试，也可以换一个角度：「我在实习里负责的是数据管道测试，验证数据在 ETL 过程中是否正确转换和加载。

这套测试框架的思路和 Agent Harness 测试是相通的——都是验证数据在经过一系列处理节点后，是否正确地流入和流出。

数据管道的边界 case 包括字段缺失、类型转换失败、外键约束违反，和 Agent 的工具调用失败场景在测试策略上高度一致。」

这个说法的本质是用类比把你已有的经验映射到目标岗位的语境里。面试官不是在考核你有没有做过一模一样的事情，而是在评估你能不能把经验迁移过来。

## 六、面试追问预测与应答策略

面试八股文的核心价值不是背标准答案，而是预判追问方向、准备好应答逻辑。

下面四个追问是测试策略相关面试里的高频出现项。每个追问后面给出了应答框架和追问预判，帮助你在面试现场建立主动权。

### 追问一：「你怎么决定哪些 case 需要加回归测试？」

这个问题本质上是在问你的优先级判断能力。

标准答法分两步。第一步，给出判断维度：「我基于改动影响面来评估优先级，优先覆盖两类 case：循环必经路径和高风险边界。」

第二步，展开两类定义。「循环必经路径指的是每次 Agent 执行任务都会经过的节点，比如状态更新、条件判断，这些路径上的改动必须覆盖完整的功能测试。

高风险边界指的是工具调用失败、上下文窗口逼近、防护栏绊线触发这类容易暴露问题的边缘条件，这些场景的错误通常是静默的，不在主流程里，但一旦触发就会导致 Agent 行为异常。」

追问预判：面试官通常会追问「高风险的定义是什么，怎么量化？」这时候要给出具体标准：「高风险操作的判断标准是：这个操作是否修改了外部状态。

比如写文件、删数据库、发送消息、调用支付接口——这些操作的漏检成本极高，阈值趋近 0%。而信息查询类操作（搜索、读取、分析）的漏检成本有限，可以适当放宽阈值。」

这个回答框架展示的不是你知道多少，而是你有一套决策逻辑。面试官要的就是这个。

> 这套框架说清楚了，面试官已经开始记笔记了

### 追问二：「如果 CI 里有一个测试间歇性失败，你怎么处理？」

这个问题在测试策略面试里几乎必问，因为它测试的是你对真实生产环境的理解。

标准答法：「首先区分 flaky test 和真实回归。Flaky test 的特征是间歇性失败，通常由网络抖动、第三方 API 不稳定或模型输出的非确定性导致。

真实回归的特征是每次都失败，且失败模式稳定可复现。」

处理方式：「对于 flaky test，我会配置重试策略，比如用指数退避的方式重试 2-3 次，如果重试后仍然失败才标记为回归问题并保留现场日志供人工排查。

对于真实回归，我会立刻定位失败原因，优先修复，最小化对主分支的阻塞时间。」

追问预判：面试官会追问「重试几次？间隔怎么定？」这时候要给出具体参数。

「重试次数通常设为 2 次，超过这个上限说明问题不是偶发的网络抖动，而是代码逻辑问题，需要人工介入。

关于间隔，业界常用指数退避，比如第一次等 1 秒，第二次等 2 秒，这样可以在不超时的情况下给系统恢复留出缓冲。

Stripe 的生产实践是把重试上限设为两次，这是一个可以直接引用的参考数据。」

### 追问三：「2% 的漏检率能接受吗？」

这道题没有标准答案，考的是判断力。

标准答法：「取决于使用场景。2% 的漏检率在信息查询场景里是可以接受的，因为错误的影响范围有限，可以通过重试自然修复。但在关键操作场景里，2% 的漏检率意味着每小时都有几次漏检，绝对不可接受。」

追问预判：面试官会追问「那怎么区分哪类是关键操作？」要给出清晰标准。「判断标准是：这个操作是否修改外部状态。

写文件、删数据库、发送消息、调用支付接口——这些操作的漏检可能造成不可逆损失，阈值必须趋近 0%。

搜索、读取、分析类操作——这些操作的漏检可以通过重试修复，阈值可以放宽到 1%-2%。」

这道题的回答质量在于：你有没有意识到漏检率不是固定数字，而是场景依赖的决策。如果你只回答「不能接受」，说明你还在「非黑即白」的思维阶段；如果你能给出判断框架，说明你已经在用工程思维做决策了。

### 追问四：「LLM 评判者的 prompt 怎么设计？」

这道题考察的是你对 LLM 评判者机制的理解深度，不是背 prompt 模板。

标准答法：「LLM 评判者的 prompt 设计需要从三个维度设定评分标准：工具调用成功率（Agent 是否在正确的时机调用了正确的工具）、任务完成率（最终是否达到了目标）、生成质量（输出是否流畅、有无幻觉）。

在 prompt 里需要明确评分标准、给出判断依据的要求，并配置 few-shot 示例让评判者学会区分不同场景。」

追问预判：面试官会追问「few-shot 示例怎么选？」要给出选样原则。「示例需要覆盖正向和负向案例。

正向案例是工具调用成功且参数正确、任务完成且质量高的样本；负向案例包括工具调用成功但参数错误、调用成功但返回异常、任务完成但输出有幻觉的样本。

这样评判者在 few-shot 学习后能更好地泛化到新样本。」

这道题的回答关键是展示你理解 LLM 评判者是一个系统，而不是一个 prompt 模板。评分维度、few-shot 示例、评判结果的解释链路，构成了一个完整的 LLM 评判系统。

## 七、易错点与高频误答警示

这一节整理了五个高频误答，每个误答后面分析了为什么危险、以及如何避免。

### 误答一：把「测试模型」说成「测试 Harness」

这是最常见的概念混淆，面试官一追问就会露馅。

危险之处在于：模型测试和 Harness 测试是不同层次的工作，面试官追问「你测的到底是什么」时，如果答不上来，就暴露了概念不清。

正确区分：模型测试测的是 LLM 的能力边界——推理质量、指令遵循、多轮对话一致性。Harness 测试测的是循环逻辑、工具编排、状态管理、错误处理这些基础设施。

TerminalBench 排名从 30 到第 5 名的案例说明，Harness 改动的效果可以比模型本身还大，这正是 Harness 重要性的来源。

### 误答二：只说「我加了测试」而不说「测了哪类边界 case」

「我为 Agent 项目加了测试」这句话在面试官耳朵里等于「我没说什么」。

追问细节时无法展开，显得经验是课程级别而非工程级别。

高质量的说法需要包含：覆盖了多少个 case、具体是哪几类边界场景（工具调用失败 / 状态合并异常 / 上下文压缩误删）、发现了什么问题、结果怎么量化。

### 误答三：把「prompt engineering」归入 Harness 测试

Harness 测试和 prompt engineering 是不同层次，但很多候选人把它们混为一谈。

Prompt engineering 是在给定 Harness 的前提下，优化模型的输入指令来改善输出质量——它的工作对象是模型。

Harness 测试是在给定 Prompt 的前提下，验证循环逻辑、状态管理、错误处理这些基础设施是否正确运行——它的工作对象是 Harness。

验证循环不是 prompt 调优。

面试官问「你测的验证循环具体测了什么」时，正确的回答是「测了规则反馈（测试 / linter / 类型检查器）、视觉反馈（Playwright 截图）和 LLM 评判者三个层次，而不是「测了 prompt 是否有效」。

![](https://iili.io/qyszWn2.png)
> 你把 prompt 调优算进 Harness 测试，reviewer 给你打回来了

### 误答四：给固定通过率数字但不解释阈值设定逻辑

很多候选人会背「我们的测试通过率是 95%」这类数字，但被追问「2% 还是 0.01%？」时答不上来。

关键在于：漏检率阈值是一个场景依赖的决策，不是一个固定数字。关键操作（删除、发送、支付）的漏检成本极高，阈值必须趋近 0%；信息查询类操作的漏检可以通过重试修复，阈值可以放宽到 1%-2%。

如果你只给数字而不给判断逻辑，面试官会认为你只做过执行层面的工作，没有参与过架构层面的决策。

### 误答五：否认没做过完整项目

「我没有做过完整的 Agent 项目」这句话说出口就输了。

正确策略是降级叙事而非否认。把课程项目里的状态机实现、实习里的数据管道测试、开源贡献里的测试 case 设计，翻译成 Harness 测试的工程语言。

诚实不等于没有替代叙事，而是把你的经验映射到目标岗位的语境里。

面试官不是在找「做过一模一样事情的人」，而是在找「能理解问题、能迁移经验、有工程判断力的人」。降级叙事做得好，比假装有过完整项目更有说服力。

### 附录 A：Harness 测试术语边界速查

| 术语 | 边界定义 | 常见混淆项 |
|---|---|---|
| **Harness** | 包裹 LLM 的完整软件基础设施：编排循环、工具、记忆、上下文管理、状态持久化、错误处理、安全防护 | 将 Harness 等同于 Prompt |
| **Agent** | 涌现行为——有目标、会用工具、能自我纠错的实体，用户看到的就是它 | 将 Agent 等同于模型 |
| **哑循环** | Harness 只管理循环回合，所有智能在模型里，不做决策推理 | 误以为 Harness 参与了 Agent 决策 |
| **验证循环** | 通过规则测试、视觉反馈、LLM 评判者验证 Agent 输出的正确性 | 与 Prompt 调优混淆 |
| **Reducer** | LangGraph 中合并节点返回值到主状态的函数，决定了状态更新的合并策略 | 与简单赋值混淆 |
| **Flaky Test** | 间歇性失败，通常由外部不稳定因素（网络、API）导致 | 与真实回归混淆 |
| **前馈验证** | 行动执行前的检查（断言、前置条件验证） | 与后馈验证混淆 |
| **后馈验证** | 行动执行后的检查（测试、截图、LLM 评判） | 与前馈验证混淆 |
